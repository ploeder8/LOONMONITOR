Technische architectuur

Modular monolith op Kotlin/Spring Boot 3, met decision tables, selectief event sourcing op de PayrollRun, en CQRS voor aangifte/loonbrief/boekhouding.

Scope: deze architectuur beschrijft de volledige loonmotor (backend + database + aangiften). De Jaakie payrollcalculator rekent browser-only in TypeScript/React/Vite; de optionele AI-chat gebruikt een aparte Vercel serverless laag — zie Jaakie-specificatie voor de actuele architectuur.

1. Architectuurprincipes

  1. Modular monolith over microservices voor fase 1–4: één deployable, duidelijke modulegrenzen, geen netwerk-overhead op de berekening. Splitsen pas bij > 50 dev's of duidelijke schaalgrens. Hypothese
  2. Decision tables als data — BV-schalen, RSZ-tarieven, werkbonus-grenzen, PC-barema's, beslagschijven leven in PostgreSQL JSONB met geldigheidsperiode. Geen Drools-runtime. MVP
  3. Selectief event sourcing — alleen op de aggregate PayrollRun (input-events + result-event). Audit-quality zonder de hele applicatie te belasten.
  4. CQRS tussen berekening (write) en aangifte/loonbrief/boekhouding (read-models, eventual consistency).
  5. Pure functions in de berekenkern — geen I/O, deterministisch, eenvoudig te testen tegen golden master.
  6. OpenAPI 3.1 first — contract-driven, gegenereerde clients, versionering via URL (/v1/) + media-type.
  7. GDPR by design — RRN/INSZ encryptie kolom-niveau, pseudonimisatie na uitdienst-termijn, RLS multi-tenancy.
  8. Idempotent & deterministisch — zelfde input + zelfde regelversie → bit-identieke output. Validator in CI.

4. Berekenings-pijplijn

// Pseudocode — pure function per PayrollRun
fun calculate(input: PayrollInput, rules: RuleSet): PayrollResult {
  val canonical   = canonicalize(input)              // stable JSON
  val inputHash   = sha256(canonical)
  val brutoLines  = composeBruto(input, rules)       // basis + premies + VAA
  val rszBase     = if (input.statuut == ARBEIDER) bruto * 1.08 else bruto
  val rszWn       = rules.rsz.calculate(rszBase, input.context)
  val werkbonus   = rules.werkbonus.calculate(bruto, statuut, voltijds)
  val belastbaar  = bruto - rszWn + werkbonus
  val bv          = rules.bv.lookup(scale, belastbaar, gezinsCoef, kinderen)
  val bbsz        = rules.bbsz.calculate(belastbaar)
  val beslag      = rules.beslag.calculate(netto, kinderenTL, rules.schijven2026)
  val netto       = belastbaar - bv - bbsz - inhoudingen - beslag + nettoVoordelen
  val wgKost      = bruto + rsz_wg + provisieVakantie + provisieEindejaar
  return PayrollResult(
    inputHash, rules.version, bruto, rszWn, belastbaar, bv, bbsz, beslag,
    netto, rsz_wg, wgKost,
    explain = buildExplainTrace()                    // per regel: id, versie, input, output
  )
}

De kern is een pure function zonder I/O. Database-writes, event-publicatie en PDF-generatie zitten buiten de berekening — testbaar in milliseconden, deterministisch tegen golden master.

Niveau 2 — Strategy-pattern in Kotlin

Sectorspecifieke premies, exotische voordelen en complexe edge cases (mobiliteitsbudget pijlers, retroactieve indexatie) als sealed interface implementaties. Eén regel = één klasse = één test.

sealed interface PremieRegel { fun apply(c: Context): LoonComponent? }

class JaarpremiePc200 : PremieRegel {
  override fun apply(c: Context): LoonComponent? {
    if (c.pc != "200" || c.maand != Month.JUNE) return null
    return LoonComponent("JAARPREMIE_PC200",
      bedrag = BigDecimal("330.84"),
      rszPlichtig = true, bvPlichtig = true, exceptioneel = true)
  }
}

Waarom geen Drools? Drools voegt complexiteit toe (DRL-syntax, runtime-engine, debugbaarheid) zonder echte winst zolang regels deterministisch en lineair zijn. Code-driven regels zijn beter testbaar en debugbaar — Drools eventueel terug op tafel bij > 500 actieve regels. Hypothese

6. Event sourcing & CQRS — selectief

Event sourcing alleen op de aggregate PayrollRun — niet over de hele applicatie. Events:

  • PayrollRunRequested — periode + werkgever + werknemers + rule_set_version
  • PayrollLineComputed — per werknemer (idempotent)
  • PayrollRunCompleted — totaalcheck (sum bruto, sum RSZ, sum BV)
  • PayrollRunSuperseded — bij correctie / herberekening

Read-models worden async opgebouwd via outbox-pattern:

Read-modelBronConsistency
Loonbrief PDFPayrollLineComputedEventual (seconden)
DmfA-inputPayrollRunCompleted + aggregatie kwartaalEventual (uren)
Belcotax 281.10Jaar-aggregatie alle runsEventual (dagen)
SEPA pain.001PayrollRunCompleted + goedkeuringEventual (minuten, na approval)
BoekhoudjournaalPayrollRunCompletedEventual (minuten)

9. Security & compliance

  • Encryptie at-rest — TDE op disk + kolom-niveau pgcrypto voor RRN/IBAN met KMS-key (Azure Key Vault / HashiCorp Vault). Key-rotation jaarlijks. MVP
  • Encryptie in-transit — TLS 1.3 publiek; mTLS service-to-service met certificate pinning.
  • RBAC + tenant-scoping — JWT met tenant-claim → SET LOCAL app.current_tenant → RLS filtert elke query.
  • MFA verplicht — admin, HR-rol, betalingen approver. WebAuthn als sterke factor.
  • Audit-log — append-only met SHA-256 ketenhash; REVOKE UPDATE, DELETE + trigger-enforcement.
  • Pentest — jaarlijks fase 1; per release fase 4. Bug bounty fase 5. Later
  • OWASP ASVS L2 als gate voor MVP, L3 voor productie fase 4.
  • Backup — PITR PostgreSQL (5 min RPO); off-site copy versleuteld, getest restore-procedure (kwartaal).
  • DR — actief/passief regio (Belgium Central → West-Europe), RTO 4 u. MVP

11. Deployment & operations

  • CI/CD — GitHub Actions / GitLab CI; trunk-based; PR-pipeline blokkeert merge bij golden-master regressie.
  • Environments — local → ephemeral PR-preview → staging (geanonimiseerde data) → prod. Feature flags per tenant. Later
  • Migraties — Liquibase expand-and-contract; nooit rename-in-place; alle DDL is rollback-baar.
  • Observability — OpenTelemetry tracing met correlatie-id (header X-Request-Id) en tenant-id in elke span. Metrics: P50/P95/P99 per pipeline-stap, error-rate per aangifte-type.
  • Alerts — SLA-deadlines (Dimona 1e dag in/uit, DmfA laatste maand kwartaal+1, BV-doorstort 15e maand+1). PagerDuty / OpsGenie.
  • On-call — runbook per aangifte-type met rollback-procedure en RSZ/FOD contactlijst.
